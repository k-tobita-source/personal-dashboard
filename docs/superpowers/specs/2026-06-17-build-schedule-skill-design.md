# build-schedule スキル設計

作成日: 2026-06-17

## 目的

毎朝（または任意のタイミング）に 1 コマンドで、その日の **schedule レーン**を全自動で組み立てる Claude Code スキル。my-kanban の MCP サーバー経由でボードを操作する。**毎日使うほど精度が上がる**よう、学習メモファイルに好みを蓄積し、実行ごとに読み込み・追記する。

単一ユーザー・ローカル完結の前提。タイムゾーンは JST 固定。

## 前提（既存の仕組み）

- MCP サーバー `my-kanban` は実装済み。利用するツール:
  - `list_tasks(lane?)` — 全タスク or レーン絞り込み。返り値は `{ id, lane, source, title, body, startAt, endAt }`（日時は ISO 文字列 / UTC、未設定は null）。
  - `move_task(id, lane, startAt?, endAt?)` — レーン移動 + 時刻設定。`startAt`/`endAt` は三値（省略=据え置き / null=クリア / ISO=設定）。schedule へ startAt 付きで入れて endAt 省略時はサービス層が既定 60 分を補完する。
  - `update_task(id, …, startAt?, endAt?)` — 時刻の微調整に使用。
- データモデル: 単一 `task` テーブル、`lane`（inbox / schedule / in_progress / done）が状態。`source` は calendar / slack / gmail / todo。
- schedule レーンは連続タイムグリッド（15 分スナップ）。高さ = `endAt - startAt`、未設定時は既定 60 分。

## 実行モード

**全自動配置**。確認を挟まず、選別 → 配置までを一気に行い、結果を要約して報告する。フィードバックは事後にチャットで受け、学習メモへ反映する。

## 配置対象と扱い

- **固定ブロック（動かさない）**: calendar 予定（startAt あり）+ すでに schedule レーンにあるタスク全て。
- **配置候補**: inbox レーンの todo / slack / gmail。
- 候補から **重要なものだけ**選別して配置する。残りは inbox に残す（詰め込みすぎない）。

## 実行フロー

1. **学習メモを読む** — `learnings.md`（存在しなければ初期テンプレートを書き出してから読む）。
2. **盤面取得** — `list_tasks` で全タスク取得し、固定ブロック / 配置候補に仕分け。
3. **前日実績から学習（任意・軽量）** — 現盤面から、前回配置に対してユーザーが動かした / 早く done にした / inbox に戻したカードを観察し、明確な傾向があれば learnings に追記。確証が持てない場合は何もしない（ノイズを増やさない）。
4. **稼働枠を決定** — 既定 10:00–19:00、昼休み 12:00–13:00 をブロック。learnings の `# 稼働時間` で上書き。固定ブロックの隙間（空き時間）を 15 分スナップで算出。
5. **重要タスクを選別** — learnings の `# 選別基準` に従う（締切の近さ・優先語・ソース・滞留日数など）。初期値は後述。
6. **所要時間を見積もり** — learnings の `# タスク種別ごとの所要時間` があれば優先、なければ既定 60 分。
7. **隙間にパック** — learnings の `# 配置ルール`（例: 朝は集中作業、会議は午後に寄せる）に従い順序を決め、`move_task(id, lane="schedule", startAt, endAt)` で確定。
8. **要約を提示** — 組んだタイムライン（時刻・タイトル）と、inbox に残したタスク + その理由を一覧で報告。
9. **フィードバック吸収** — ユーザーが補正コメントを返したら、該当セクションに 1 行追記する。

## learnings.md

### 場所
`.claude/skills/build-schedule/learnings.md`

### git 扱い
日々更新されるため、リポジトリには **初期テンプレートのみ**コミットし、`learnings.md` 実体は `.gitignore` する。スキル初回実行時に実体が無ければテンプレートから生成する。テンプレートは `learnings.template.md` としてコミットする。

### 構造（追記しやすい固定セクション）

```markdown
# 稼働時間
- 平日 10:00–19:00 / 昼休み 12:00–13:00

# 配置ルール
- （例）朝イチ（10:00–12:00）は集中作業を優先で置く

# タスク種別ごとの所要時間
- （例）コードレビュー = 30分

# 選別基準
- 締切が今日/明日のものを最優先
- slack/gmail 由来は当日中に一次対応
- inbox 滞留が長いものを優先的に消化

# やらないこと
- （例）金曜午後はミーティングを入れない
```

各セクションは Claude が 1 行ずつ追記する。矛盾する古い行は上書き/削除してよい。

## 選別基準の初期値

learnings 未整備時のフォールバック:

1. 締切・日付の手がかり（body/title 内の「今日」「明日」「期限」等）が近いもの。
2. slack / gmail 由来（依頼・問い合わせ）は当日一次対応として優先。
3. 同条件なら inbox 滞留が長い（古い）ものを優先。
4. 稼働枠に収まる範囲で上位を採用し、あふれた分は inbox に残す。

## エッジケース

- **重要タスク過多** → 稼働枠に収まる上位のみ配置。残りは inbox に残し、理由（時間不足）を報告。
- **空き時間なし** → 配置せず「本日は固定予定で埋まっています」と報告。
- **配置候補ゼロ** → 「inbox に今日やるべきタスクはありません」と報告。
- **タイムゾーン** → 稼働枠（10:00 等）は JST で解釈し、`move_task` には JST を表す ISO 文字列（オフセット付き）を渡す。`list_tasks` 返り値の UTC ISO も JST に直して突き合わせる。

## スコープ外（YAGNI）

- 確認ダイアログ / 段階配置モード（全自動のみ）。
- 複数日・週次プランニング（当日のみ）。
- カレンダーへの書き戻し（ボード内で完結）。
- MCP ツールの新規追加（既存ツールのみで実現）。
